# Phase 4: Automated PRs

## Context

Phase 3 gives the operator approve/reject control over consolidation proposals, with mutations applied to D1 tables at runtime. Phase 4 adds an optional code-level audit trail: when proposals are approved, the system opens a PR that commits the corresponding config-file changes to the repo. This means runtime state (D1) and source-of-truth (git) stay in sync without the operator having to manually edit files.

Phase 4 is the most risk-sensitive phase. A bug here writes to the repo. The scope is deliberately narrow and every safeguard is documented.

## Prerequisites

- Phase 3 has been running for 4+ weeks
- The operator has approved 10+ proposals and is confident in the proposal quality
- The D1 runtime-override pattern is working reliably
- A GitHub fine-grained PAT is available with the minimum required scopes

## Hard Scope Constraints

### Files the PR system is allowed to modify

```
src/scoring/thresholds.ts
src/scoring/topic-hints.ts
src/scoring/prompt-config.ts
```

That's it. Three files. All are structured data/config — no business logic, no control flow, no imports.

### Files the PR system is NEVER allowed to modify

- `src/scoring/classifier.ts` — permanently off-limits
- `src/workflow.ts` — pipeline orchestration
- `src/index.ts` — request routing
- `src/ingest.ts` — ingest orchestration
- Any migration file — schema changes require human review
- Any test file — test changes are a human-judgment call
- `CLAUDE.md` — project instructions
- `wrangler.toml` — deployment config

### The PR system does NOT

- Auto-merge PRs. Every PR requires human review and merge.
- Push to main. PRs are opened against main from a branch.
- Edit files that aren't in the allow-list. The write function rejects paths outside the allow-list before touching the filesystem.
- Run without a human having approved the underlying proposals in Phase 3. PRs are opened FROM approved proposals, not FROM raw AI output.

## Architecture

```
Phase 3: operator approves proposal
  └─ D1 mutation applied (runtime effect is immediate)
  └─ If auto-PR is enabled:
       └─ PR creation job
            ├─ Read the approved proposal + current file content
            ├─ Compute the diff (what the config file should look like
            │   given the current D1 override state)
            ├─ Validate the path is in the allow-list
            ├─ Create branch, commit, open PR via GitHub REST API
            └─ Record PR URL in proposal_actions
```

### Two-phase commit model

The D1 mutation is the primary write (Phase 3). The PR is a secondary, asynchronous audit trail. If the PR fails to open, the runtime mutation is already live — the operator just needs to manually sync the config file later. This means:

- PR creation failure is non-blocking
- The runtime is never dependent on the PR being merged
- If the operator closes the PR without merging, the D1 override still applies
- If the operator wants to undo, they delete the D1 override row AND close the PR

The PR exists so that `git log` matches runtime state, not so that runtime state depends on `git log`.

## GitHub Integration

### Authentication

A GitHub fine-grained Personal Access Token (PAT), scoped to:
- **Repository:** this repo only
- **Permissions:** Contents (read + write), Pull Requests (read + write)

Stored as `GITHUB_TOKEN` in Worker secrets. The repo identifier is stored as `GITHUB_REPO` (e.g., `delder004/aggregator`).

### PR creation flow

All via the GitHub REST API using `fetch` — no Octokit dependency.

1. `GET /repos/{owner}/{repo}/git/ref/heads/main` — get main's current SHA
2. `POST /repos/{owner}/{repo}/git/refs` — create branch `consolidation/{consolidationId}-{timestamp}`
3. `GET /repos/{owner}/{repo}/contents/{path}?ref=main` — get current file content + SHA
4. `PUT /repos/{owner}/{repo}/contents/{path}` — commit the updated file to the branch
5. `POST /repos/{owner}/{repo}/pulls` — open PR with:
   - Title: `[Consolidation] {proposal.action} {proposal.target}`
   - Body: proposal rationale + link to `/ops/consolidations/{id}` + link to `/ops/proposals/{id}/{index}`
   - Base: `main`
   - Head: the branch
   - Labels: `consolidation-proposal` (created if it doesn't exist)

### PR body template

```markdown
## Consolidation Proposal

**Type:** {type} | **Action:** {action} | **Confidence:** {confidence} | **Priority:** {priority}

**Target:** {target}

**Rationale:** {rationale}

## Change

{diff preview}

## Context

- Consolidation: [{consolidationId}](/ops/consolidations/{consolidationId})
- Week: {windowStart} to {windowEnd}
- Approved at: {actedAt}

---
Generated by the weekly consolidation loop. Review before merging.
```

## Config File Format

Each allow-listed config file has a machine-readable format:

### `src/scoring/thresholds.ts`

```typescript
// Auto-generated from D1 scoring_overrides. Do not edit manually.
// Manual edits will be overwritten by the next consolidation PR.

export const THRESHOLDS = {
  MIN_PUBLISH_SCORE: 50,
  FEATURED_SCORE: 70,
  MAX_SCORE_PER_RUN: 50,
} as const;
```

### `src/scoring/topic-hints.ts`

```typescript
// Auto-generated from D1 topic_hints. Do not edit manually.

export const TOPIC_HINTS: Record<string, number> = {
  "ai tax preparation": 1.2,
  "agentic ai audit": 1.1,
};
```

### `src/scoring/prompt-config.ts`

```typescript
// Auto-generated from D1 prompt_overrides. Do not edit manually.

export const CLASSIFIER_PROMPT_ADDITIONS: string[] = [
  "Boost articles about AI agents completing end-to-end tax workflows.",
  "Deprioritize generic AI newsletter roundups unless they mention accounting.",
];
```

The PR creation function reads the current D1 override state, renders the entire file, and commits it. This is a full-file write, not a line-level patch — simpler, more predictable, and the diff viewer shows exactly what changed.

```
GITHUB_TOKEN    — fine-grained PAT (Contents + Pull Requests, this repo only)

```

## Collision Handling

**If a consolidation PR is already open when a new one fires:**

- Check for open PRs with the `consolidation-proposal` label
- If one exists and is less than 7 days old: skip PR creation, log a warning, record `pr_status: 'blocked'` in the proposal action
- If one exists and is more than 7 days old: close the stale PR with a comment ("superseded by newer consolidation"), then open the new one
- This prevents PR spam while ensuring stale proposals don't block progress

## Ops Endpoints

```
POST /ops/proposals/:consolidationId/:index/open-pr   — manually trigger PR for an approved proposal
GET  /ops/prs/pending                                  — list open consolidation PRs
```

These supplement the Phase 3 endpoints. The `open-pr` endpoint is separate from `approve` so the operator can approve without opening a PR (useful for proposals that only need a D1 mutation, like adding a source).

## Commit Plan

1. **GitHub REST client** — raw `fetch`-based client for branch creation, file commit, PR opening. Path allow-list enforcement. No Octokit.
2. **Config file renderer** — reads D1 override state, renders each allow-listed file as a complete TypeScript module.
3. **PR creation flow** — branch → commit → PR with templated body. Collision detection.
4. **Wire into Phase 3 approve flow** — optional PR creation after D1 mutation. New `/ops/proposals/*/open-pr` endpoint.
5. **Stale PR cleanup** — auto-close PRs older than 7 days when a new consolidation fires.

## Success Criteria

1. Approving a `threshold/adjust` proposal writes to D1 AND opens a PR that modifies `thresholds.ts`
2. The PR diff shows exactly the threshold change, nothing else
3. Merging the PR makes `git log` match the runtime D1 state
4. Closing the PR without merging has no runtime effect (D1 override still applies)
5. Failed PR creation does not block or roll back the D1 mutation
6. PRs are only opened for the 3 allow-listed files — any attempt to modify other files is rejected before the GitHub API is called
7. Stale PRs are auto-closed when superseded

## Cost Estimate

GitHub API calls per proposal PR: ~5 (ref lookup, branch creation, file read, file commit, PR creation). Well within GitHub's rate limits. No additional Anthropic API cost — the AI reasoning happened in Phase 2's consolidation step.

## Risk Mitigations

| Risk | Mitigation |
|---|---|
| PR modifies wrong file | Path allow-list checked before GitHub API call; rejection is a thrown error, not a silent skip |
| PR contains bad code | Files are structured data with a fixed template; the renderer doesn't accept arbitrary code |
| Stale PRs pile up | Auto-close after 7 days when superseded |
| GitHub token leak | Fine-grained PAT scoped to one repo, Contents + PRs only; stored in Worker secrets |
| Operator confusion about runtime vs. git state | PR body explicitly states "D1 mutation already applied; this PR syncs the repo" |
| Double PR for same proposal | `proposal_actions` unique index prevents double-approve; PR creation checks for existing open PRs |
