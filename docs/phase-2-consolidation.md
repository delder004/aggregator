# Phase 2: Weekly Consolidation Dry-Run

## Context

Phase 1 (commits 1-7) deployed five capture namespaces that collect data weekly:

| Namespace | What it captures | Window |
|---|---|---|
| cf-analytics | Zone totals: requests, pageViews, uniqueVisitors, bytes, cachedRequests | Previous week |
| search-console | Authoritative totals (dimensions:[]) + top queries + top pages | Previous week (PT-aligned) |
| rankings | Serper rank for 20 seed keywords | Current week |
| competitors | RSS/homepage items for 8 competitors | Previous week |
| article-views-rollup | Per-article daily views from Analytics Engine | Last 7 complete UTC days |

Phase 1 also has per-run AI retrospectives on every hourly pipeline run (stored in `pipeline_runs.retrospective_*` columns, generated by `src/runs/retrospective.ts` via Haiku).

Phase 2 reads all of that, feeds it to a single Sonnet call, and stores the output for inspection. No mutations, no PRs, no code changes to the pipeline. Pure read + reason + persist.

## Goal

Build and inspect the weekly consolidation output for 2-3 weeks before Phase 3 adds any mutation capability. Success criteria:

1. `assembleConsolidationContext()` produces a token-budgeted prompt from all six input families
2. A single Sonnet call returns structured proposals
3. The raw context, AI output, and parsed proposals are persisted and inspectable via `/ops/consolidations[/:id]`
4. Dry-run output is reviewed by the operator each week — no automated action taken

## Data Model

### New table: `run_consolidations`

```sql
CREATE TABLE IF NOT EXISTS run_consolidations (
    id TEXT PRIMARY KEY,
    window_start TEXT NOT NULL,
    window_end TEXT NOT NULL,
    status TEXT NOT NULL DEFAULT 'running',
    attempt_count INTEGER NOT NULL DEFAULT 0,
    started_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,
    completed_at TEXT,
    error_message TEXT,

    -- Input refs: which snapshots/runs were read
    input_run_ids_json TEXT NOT NULL DEFAULT '[]',
    input_snapshot_ids_json TEXT NOT NULL DEFAULT '{}',

    -- Large payloads in KV, queryable metadata in D1
    context_blob_key TEXT,
    context_token_estimate INTEGER,

    -- AI output
    ai_model TEXT,
    ai_output_blob_key TEXT,
    ai_summary TEXT,
    ai_proposals_json TEXT,
    ai_token_usage_json TEXT
);

CREATE UNIQUE INDEX IF NOT EXISTS idx_run_consolidations_window
    ON run_consolidations(window_start, window_end);
```

- Same claim/complete/fail + attempt_count pattern as the snapshot tables.
- `context_blob_key` and `ai_output_blob_key` point to KV blobs under `consolidations/<windowStart>/<id>.json` — the full prompt and raw AI response stay out of D1.
- `ai_proposals_json` is a small structured array kept in D1 for queryability.
- `ai_summary` is a 1-2 sentence overview kept in D1 for the list endpoint.

## Architecture

```
Weekly cron (Mon 13:00 UTC)
  └─ IngestWorkflow (Phase 1 — already deployed)
       └─ after all 5 namespaces complete:
            └─ ConsolidationStep (Phase 2 — new)
                 ├─ assembleConsolidationContext()
                 ├─ callSonnet()
                 ├─ parseConsolidationResponse()
                 └─ persist to run_consolidations + KV
```

### Option A: New step in IngestWorkflow

Add a 6th `step.do('consolidation', ...)` at the end of `IngestWorkflow.run()`. It only fires if at least 3 of 5 namespaces completed successfully (partial data is better than no consolidation, but total capture failure should skip).

### Option B: Separate ConsolidateWorkflow

Independent Cloudflare Workflow with its own cron (e.g., `0 15 * * 1` — 2 hours after ingest). Reads from D1/KV without depending on the IngestWorkflow's lifecycle.

**Recommendation: Option A.** The consolidation reads the same week's data that the ingest just wrote. A separate workflow would need to either (a) blindly assume ingest is done (fragile) or (b) poll ingest_runs for completion (complex). Adding a step to IngestWorkflow keeps the dependency explicit and the ordering guaranteed. If the consolidation step fails, the ingest results are already persisted — the failure is isolated.

## Context Assembly

### Input families and per-family budgets

The consolidated prompt has six input families. Each is pre-aggregated before serialization to stay within the Sonnet context window. Hard caps per family prevent prompt sprawl as data accumulates.

| Input family | Source | Cap | What gets sent |
|---|---|---|---|
| **Pipeline run summaries** | `pipeline_runs` + `pipeline_run_steps` (last 7 days) | 10 runs max | Per-run: status, step metrics, retrospective summary. NOT individual step details. |
| **CF analytics** | `cf_analytics_snapshots` + KV blob (this week) | 1 snapshot | Totals only (requests, pageViews, uniqueVisitors, cachedRequests). |
| **Search Console** | `search_console_snapshots` + KV blob (this week) | Top 20 queries, top 10 pages | Sorted by impressions DESC. Include clicks, CTR, avg position per row. |
| **Rankings** | `keyword_rankings` (this week + previous week) | All keywords (max 30) | Current rank, previous rank, rank delta. Flag keywords near page 1 (rank 8-12). |
| **Competitors** | `competitor_snapshots` + KV blobs (this week) | Top 10 items per competitor, max 5 competitors | Direct-bucket only. Item titles + URLs. |
| **Article views** | `article_views` (this week) | Top 20 articles by views | Join with `articles` table for title + score + tags. Flag mismatches: high-score/low-views, low-score/high-views. |

### Token budget

Target: ~15K input tokens for the assembled context. Sonnet 4 has a 200K context window; 15K leaves ample room for the system prompt and output without approaching limits or paying for unnecessary input.

Pre-aggregation is the defense: we send rolled-up summaries, not raw data. The `contextTokenEstimate` field in `run_consolidations` tracks actual prompt size for monitoring — if it trends past 20K, the operator should tighten per-family caps before it reaches 30K.

### Missing query: top articles by views

Phase 1 intentionally omitted `listTopArticleViews()` because the correct shape wasn't known yet. Phase 2 needs:

```sql
SELECT a.id, a.title, a.headline, a.relevance_score, a.tags,
       SUM(av.views) AS total_views
FROM article_views av
JOIN articles a ON av.article_id = a.id
WHERE av.view_date >= ? AND av.view_date < ?
GROUP BY av.article_id
ORDER BY total_views DESC
LIMIT ?
```

This returns one row per article with aggregated weekly views — the correct shape for the consolidation prompt.

## AI Call

### Model

`claude-sonnet-4-6-20250514` (Sonnet 4.6). The retrospective uses Haiku because it reasons over a single run's structured facts. The consolidation reasons across six input families, compares trends, and generates proposals — that requires Sonnet-class reasoning.

### System prompt

```
You are a strategic analyst for agenticaiaccounting.com, an automated
news aggregator covering agentic AI in accounting.

You will receive a weekly context bundle containing:
- Pipeline run summaries with AI retrospectives
- Cloudflare zone analytics (traffic totals)
- Google Search Console data (queries, pages, impressions, clicks, CTR)
- Keyword ranking positions and week-over-week deltas
- Competitor content snapshots (what they published this week)
- Top articles by views with their relevance scores and tags

Analyze all inputs together and return valid JSON with this schema:
{
  "summary": "<2-3 sentence overview of this week>",
  "whatWorked": ["<item>", ...],
  "whatDidnt": ["<item>", ...],
  "proposals": [
    {
      "type": "<source | threshold | topic | keyword | competitor>",
      "action": "<add | remove | adjust | investigate>",
      "target": "<specific thing to change>",
      "rationale": "<why, citing specific data from the context>",
      "confidence": "<high | medium | low>",
      "priority": "<high | medium | low>"
    }
  ],
  "topicGaps": ["<topic competitors cover that we don't>", ...],
  "keywordOpportunities": ["<keyword where we're close to page 1>", ...]
}

Rules:
- Be concrete. Cite specific numbers, article titles, keyword ranks,
  or competitor items when making a claim.
- Each proposal must reference data from the context. No speculative
  proposals without supporting evidence.
- Keep proposals to 3-8 items. Quality over quantity.
- "whatWorked" and "whatDidnt" should each have 2-4 items.
- "topicGaps" and "keywordOpportunities" can be empty if the data
  doesn't support them.
- Proposal types:
  - source: add/remove/investigate a content source
  - threshold: adjust scoring thresholds or featuring logic
  - topic: boost/deprioritize a topic area
  - keyword: add/remove/adjust a tracked keyword
  - competitor: add/remove a tracked competitor
```

### Token limit

2048 max output tokens. The structured JSON response with 3-8 proposals, 2-4 items per list, and citations fits within ~1500 tokens; the extra headroom prevents truncation of longer rationale strings.

### Response parsing

Same pattern as `parseRunRetrospectiveResponse()`:
1. Strip markdown code fences via `extractJsonObject()`
2. Parse JSON
3. Validate required fields are present and non-empty
4. Type-narrow arrays and strings
5. Throw on invalid structure (consolidation marked as failed, operator inspects)

### Cost estimate

One Sonnet call per week: ~15K input tokens + ~1.5K output tokens. At Sonnet pricing ($3/M input, $15/M output): ~$0.045 + ~$0.023 = **~$0.07/week**, or ~$3.50/year. Negligible.

## Success Criteria

1. Weekly consolidation runs complete and produce inspectable output at `/ops/consolidations/:id`
2. The AI summary and proposals reference specific data from the context (not generic platitudes)
3. `contextTokenEstimate` stays under 20K
4. Proposals are actionable: an operator reading them can decide yes/no without needing to re-research
5. After 2-3 weeks of inspection, proposals are consistently useful enough to justify Phase 3 (human-approved mutations)
